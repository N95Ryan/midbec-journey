# 📅 Phase 1 — Alignement catalogue UnoPim (local)

## Contexte

Patrick confirme la nouvelle architecture : **UnoPim = source de vérité produits**, **Ogasys/ERP = tout le reste** (prix, stock, comptes, commandes, factures). Phase 1 planifiée : valider OAuth UnoPim via smoke test, basculer `NEXT_PUBLIC_CATALOG_SOURCE=pim` en local, QA catalogue.

## Ce qui a été fait

**Smoke test PIM (`pim-smoke.sh`) — matin**

- API Go déjà en cours d'exécution (`go run ./cmd`, port 8080)
- Commande : `./scripts/pim-smoke.sh http://localhost:8080 80040`
- **Résultat initial : Branche B** — OAuth UnoPim down (message générique `status 500`)

**Diagnostic OAuth approfondi — après-midi**

- Nouveau script : `go run ./scripts/pim-oauth-probe.go ./scripts/probeenv.go`
- Serveur UnoPim joignable : `GET http://10.10.10.6:8000/` → 200
- **Cause racine identifiée** (identique sur les 4 stratégies d'auth testées) :

```json
{
  "message": "Key file \".../Secrets/Oauth/oauth-private.key\" permissions are not correct, recommend changing to 600 or 660 instead of 664",
  "exception": "ErrorException",
  "file": "/var/www/html/vendor/league/oauth2-server/src/CryptKey.php"
}
```

- Ce n'est **pas** un bug du client Go ni des credentials `.env` — c'est un fix **one-liner** côté serveur UnoPim (Patrick).

**Correctif serveur (Patrick / SSH sur `10.10.10.6`)**

```bash
chmod 600 /var/www/html/packages/Webkul/AdminApi/src/Secrets/Oauth/oauth-private.key
chmod 600 /var/www/html/packages/Webkul/AdminApi/src/Secrets/Oauth/oauth-public.key
# puis vérifier :
go run ./scripts/pim-oauth-probe.go ./scripts/probeenv.go   # depuis midbec-go-api
./scripts/pim-smoke.sh http://localhost:8080 80040
```

**Amélioration code Go**

- `internal/clients/unopim/client.go` — le corps de réponse Laravel est inclus dans l'erreur OAuth (plus de `status 500` silencieux)
- `docs/unopim-catalogue.md` — section 9 dépannage OAuth ajoutée

**Bascule catalogue — non effectuée (gate Branche B)**

Tant que `POST /oauth/token` → 500, le front reste sur `erp` ([`midbec-front/.env.local`](../../midbec-front/.env.local)).

**QA API (mode ERP actuel — non-régression)**

| Endpoint | HTTP | Statut |
|----------|------|--------|
| `GET /api/categories/tree` | 200 | OK |
| `GET /api/search?q=80040` | 200 | OK |
| `GET /api/products/80040` | 200 | OK (panier/prix) |
| `GET /pim/categories/tree` | 502 | Bloqué OAuth (permissions clé) |
| `GET /pim/products/80040` | 502 | Bloqué OAuth (permissions clé) |

## Ce que j'ai appris

- Un HTTP 500 OAuth n'est pas toujours « Passport cassé » — ici c'est une **permission fichier** (`664` au lieu de `600`), beaucoup plus simple à corriger.
- Tester plusieurs formats d'auth (form, JSON, Basic) permet d'écarter un bug client Go : toutes les stratégies échouent pareil → problème serveur.
- Inclure le body Laravel dans les erreurs Go économise des heures de debug futur.

- Après fix : re-smoke → bascule `NEXT_PUBLIC_CATALOG_SOURCE=pim` + QA UI.
- Import SKU UnoPim ↔ ERP : Branche A_partial possible même après OAuth OK.
