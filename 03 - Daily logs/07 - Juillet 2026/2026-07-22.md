## 📅 Mercredi 22 Juillet 2026 — Factures : nommage des téléchargements

### 🧠 Humeur & contexte

Suite à la clôture du chantier factures (download + filtre impayées), retour UX : les clients doivent retrouver leurs PDF facilement après téléchargement. Ajustement du nommage des fichiers pour qu'il soit explicite, que ce soit un PDF seul ou plusieurs factures dans un ZIP.

### 🎯 Objectifs

- [x] Téléchargement d'une seule facture → PDF direct (plus de ZIP inutile)
- [x] Format de nom uniforme : `Facture - #<numéro>.pdf`
- [x] Même format pour les entrées internes du ZIP (sélection multiple)
- [x] Tests automatisés sur le nommage
- [x] Commit + push api et journal

### 💡 Découverte du jour

**Un ZIP pour une seule facture, c'est une friction inutile.** Comme recevoir une enveloppe scellée pour un seul document — le client doit extraire le PDF pour rien.

**Le nom du fichier est une partie de l'UX.** Sans libellé clair, la comptabilité renomme à la main ou ouvre chaque PDF pour identifier la facture.

### ✅ Réalisations

- `DownloadInvoices` : 1 facture → réponse `application/pdf` directe
- Nom partagé via `invoiceFilenameStem` : `Facture - #04F1015696.pdf`
- ZIP multi-sélection : entrées internes au même format (doublons → `Facture - #…_1.pdf`)
- Tests : `TestInvoiceDownloadFilename`, `TestInvoiceZipEntryName` mis à jour
- Doc : `API_DOCUMENTATION.md`, `erp-map.md`

### 🧠 Ce que j'ai appris

- Les tests automatisés verrouillent des règles simples (nom de fichier) que la validation manuelle oublie facilement en refactor.
- Centraliser le nommage (`invoiceFilenameStem`) évite la divergence PDF direct vs ZIP.

### ⚠️ Vigilance

- Caractères interdits Windows (`/`, `\`, `:`, etc.) toujours sanitizés dans le numéro de facture
- Redémarrer l'API Go après déploiement pour prendre en compte le changement
